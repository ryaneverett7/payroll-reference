What Is an API, Really?

Most people hear "API" and think: data moving between systems. That's not wrong, but it misses the most important part. An API is a governed contract — it defines what is allowed, under what conditions, and what happens when rules are broken.

🔑 Core Concept

An API is not a pipe that moves data. It is a contractual boundary between systems that exposes capabilities. Every single API call is inspected for identity, permission, validity, and context. If any of those checks fail, the call is rejected.

This distinction matters because it changes how you think about problems. When a partner says "the data didn't update," you don't think the pipe is broken — you think which rule wasn't satisfied?

🍽️ Analogy — Ordering at a Restaurant

You don't walk into the kitchen, grab ingredients, and cook your own meal. You interact through a menu and a server — that's the API.

RestaurantAPI Equivalent
MenuAPI documentation
Menu itemsAvailable resources / endpoints
Server (waiter)The API interface
KitchenThe backend system
Your order slipThe API request
The meal servedThe API response

You can only order what's on the menu. The server checks if your order is allowed right now. And when things go wrong, the rejection is informative: "We're out of that" = resource not found. "That's not on the menu" = invalid request.

The Request–Response Cycle

Every API interaction follows the same three-step loop. Everything else in API design is built on top of this cycle.

1
Ask

A client sends a request — it specifies what it wants and proves it has the right to ask.

2
Process

The server inspects the request: Is it valid? Is the caller who they say they are? Are they allowed to do this?

3
Answer

The server responds — with data if successful, or a status code explaining what went wrong if not.

💡 Why this matters at EN

Employee Navigator is a hub, not a data dump. Hubs require rules and intentional "trust" boundaries. When a partner integrates with EN, every single call goes through this cycle — which is why understanding it is the foundation of everything else in this training.

✏️ Check Your Understanding

✅ Correct! An API enforces rules on every interaction — it's a governed contract, not a passive pipe. This mental model is what separates people who truly understand APIs from those who just use them.

Statelessness — APIs Remember Nothing

APIs are purposefully stateless. Think of talking to a goldfish — there is no "last time," no "we already talked," and no "you were here before." Every single request must stand completely on its own.

🔑 Core Concept

Statelessness means every API request must fully re-establish identity, permission, and intent. The server does not retain anything from previous requests. Authentication and authorization happen fresh on every call.

Why is this by design?

Partners often push back: "But we already authenticated earlier — why do we need to do it again?" The answer is that statelessness isn't a bug. It's what makes APIs work reliably at scale.

Scalability

  • Any server can handle any request
  • No shared memory required between servers
  • Systems grow without complexity exploding

Security

  • Less sensitive data stored server-side
  • Access can be revoked at any time
  • Fewer attack surfaces to protect

Predictability

  • Same input → same evaluation, every time
  • No hidden "session context" that surprises you
  • Easier to debug and reproduce issues

Access Tokens

  • Tokens exist to mitigate the burden of re-auth
  • Short-lived credentials that prove identity
  • Partners obtain them once, use for a window
⚠️ Common Partner Misconception

When a partner says "We already authenticated," the correct response is: there is no "already" in a stateless API. Their token may still be valid, but every request must include it. If a call fails with a 401, their token has expired and they need to re-authenticate to get a new one.

✏️ Check Your Understanding

A partner says "We authenticated 10 minutes ago — why is the API returning 401 Unauthorized?" What is the most likely explanation?

AThe API remembered their previous session but the server restarted
BThey need to call the API from a different IP address
CTheir access token has expired — APIs are stateless and every request must include a valid token
DThe API is down and timing out
✅ Correct! APIs never remember prior sessions. Tokens are short-lived credentials — when they expire, a new one must be obtained. A 401 means the token is missing, invalid, or expired.

Module 3

HTTP & Payloads — How APIs Talk

HTTP is the language systems use to communicate over the web. It's not just about pushing data through a wire — every API call uses HTTP to express what someone is acting on, what they want to do, how they're allowed to do it, and what data is involved.

The 5 HTTP Methods You Need to Know

MethodPlain EnglishExample at ENRisk Level
GET"Show me"Fetch an employee's demographic dataLow
POST"Create this"Add a new employee deductionHigh
PUT"Replace this entirely"Replace a full employee profileHigh
PATCH"Change part of this"Update only an employee's deduction amountMedium
DELETE"Remove this"Delete a deduction recordVery High

Anatomy of an API Request

Every API request has four components working together. Here's a real EN example — fetching agency management data:

URL
What you're acting on

https://qa.employeenavigator.net/api/companies/AMS123/agency-management — this is the address of the specific resource.

MTH
What you want to do

GET — we're reading data, not writing it.

HDR
How you're allowed to do it

Headers carry the access token and other metadata that prove identity and intent. We cover these in the Authentication module.

BDY
What data is involved

For a GET request, the body is blank — we're just asking for something to be returned. For a POST or PATCH, the body contains the data being sent.

JSON — The Shared Language

APIs use JSON (JavaScript Object Notation) to structure the data they exchange. It's a format of key-value pairs that both systems understand.

Example JSON Payload — Employee Deduction
{
  "employeeId": "EMP-10042",
  "companyId": "AMS123",
  "deductionCode": "MDCL",
  "effectiveDate": "2025-01-01",
  "perPayAmount": 142.50,
  "taxStatus": "PreTax"
}
💡 Key → Field Mapping

In JSON, each key corresponds to a field in both systems. When EN sends firstName, the payroll partner maps that key to their own first name field. If the key is missing or misspelled, the payload fails validation — which is why exact field matching matters in every integration.

✏️ Check Your Understanding

EN needs to send a new benefit deduction for an employee to the payroll partner. Which HTTP method should EN use?

AGET — to retrieve the current deduction amount from the partner first
BDELETE — to remove the old amount before adding the new one
CPOST or PATCH — POST to create a new deduction record in payroll, PATCH to update part of an existing one
DPUT — to replace the entire employee record including the deduction
✅ Correct! Deductions flow one direction: EN sends them TO the payroll partner — the partner never sends deductions to EN. POST creates a new deduction record in payroll, PATCH updates part of an existing one. GET is read-only and never changes data.

Module 4

Authentication vs. Authorization

These two words are used interchangeably all the time — but they are completely separate concepts that happen at different times for very different reasons. Mixing them up is one of the most common sources of confusion in API troubleshooting.

Authentication — "Who are you?"

  • Happens once — when a token is issued
  • Proves identity (credentials, MFA, SSO)
  • Result: an access token
  • Knowing who you are doesn't mean you can do anything yet

Authorization — "What can you do?"

  • Happens on every API call
  • Evaluates permission for this specific resource
  • Contextual — same identity may be allowed Plan A but not Plan B
  • Checked even if you've been authenticated 100 times before
🎤 Analogy — The Concert

Authentication: You show your ticket at the gate. Security confirms it's real. You're authenticated — you've proven your identity.

Authorization: Once inside, security checks: Is this ticket for today? For this venue? For floor seats or balcony? Even though you're authenticated, your specific permissions are evaluated at every door.

Authentication gets you in the building. Authorization decides how far you can go.

The API's Decision Tree

Every API call runs through a layered set of checks. Here's what EN (or any partner system) evaluates on every request:

1
Is there authentication information?

If no token is present → 401 Unauthorized. The request is rejected immediately.

2
Is the token valid?

If the token is expired, forged, or malformed → 401 Unauthorized. Identity could not be confirmed.

3
Is this identity allowed to call this endpoint?

If the identity doesn't have permission for this endpoint → 403 Forbidden. "We know who you are, but you can't do this."

4
Is this identity allowed to act on this specific resource?

Even if the endpoint is allowed, access to a specific company or employee may not be → 403 Forbidden.

✏️ Check Your Understanding

A partner receives a 403 Forbidden response when trying to fetch plan data. What does this tell you?

ATheir token is expired and they need to re-authenticate
BThe API is down or unavailable right now
CThey are authenticated (EN knows who they are) but they don't have permission to access this specific resource
DTheir JSON payload is malformed or missing required fields
✅ Correct! 403 means identity is confirmed but permission is denied. A 401 would indicate an authentication problem. Knowing the difference lets you direct partners to the right fix immediately.

Module 5

Identity Context — System vs. User

Before diving in, ask yourself: "Who is actually performing this action?" Is it a system acting on its own, or a system acting on behalf of a specific user? That distinction changes the risk profile of every API interaction.

🔑 The Bottom Line

Client Credentials = "the system decided" this was OK.
Authorization Code = "a person decided" this was OK.
Both flows exist because identity context changes risk.

Flow 1: Client Credentials — System Acting Independently

Used when two systems need to sync data automatically, with no user involved. EN's payroll integrations use this flow — a partner system authenticates itself and EN issues a token representing the application.

1
Partner system authenticates itself

Uses its client ID and secret — no browser, no login screen, no user.

2
EN issues a token representing the application

One token for the whole application. Broad, consistent permissions.

3
Application uses that token to call APIs

All calls are made as the application itself. No individual user is accountable.

⚠️ Common Misuse

"Why don't we just use Client Credentials for everything?" — Because there is no user accountability and it risks over-permissioning. A system acting independently has broad access. If that token is compromised, the blast radius is large. Reserve this for true background/sync jobs.

Flow 2: Authorization Code — System Acting on Behalf of a User

Used when user consent and accountability matter. EN's AMS (Agency Management System) integrations use this — a broker explicitly authorizes access to their book of business.

1
The user authenticates with EN

They log in and confirm their identity.

2
User consents to the access

"Yes, I authorize this system to act on my behalf."

3
EN issues an authorization code on the user's behalf

The system exchanges that code for a token representing user-delegated authority.

4
The token represents user-delegated authority

If the user can't do it in the UI, the API won't allow it either. The permissions are bounded by what that user is allowed to do.

Side-by-Side Comparison

AspectClient CredentialsAuthorization Code
Who is acting?Application itselfUser via application
User involved?NoYes
Permissions based onApp configurationUser's own rights
Audit trailSystem activityUser activity
Best forBackground jobs, payroll syncUI-driven actions, AMS access
Risk if misusedOverreach / over-permissioningLeakage of user authority
✏️ Check Your Understanding

A partner asks: "Why can't our system automatically pull broker data without requiring the broker to log in?" What is the correct explanation?

AEN's systems don't support automated access at all
BAccessing a broker's book of business requires the Authorization Code flow, which requires user intent and consent — the broker must actively authorize it
CThe Client Credentials flow could be used instead if they have the right scope
DThey need a higher-tier API subscription to enable automated access
✅ Correct! The Authorization Code flow exists because some actions require user intent. A person must actively decide to grant access. You can't automate that away — and you shouldn't want to, because it protects the broker's data.

Scopes — Asking for Permission

Scopes don't grant access — they ask for it. A scope says "I would like to do this kind of thing." The API still decides whether to allow it. Think of them as permission requests, not permission grants.

🔑 Core Concept

Scopes solve three problems: they limit risk (a compromised token can only do specific things), they express intent clearly (the API knows what kind of access is being requested), and they enable layered authorization (scopes are checked in addition to identity).

A Real EN Example

The endpoint /companies/{companyId}/agency-management returns highly sensitive data: employee demographics, dependents, eligible benefits, and enrolled plans.

Because of that sensitivity, EN requires a scope of AgencyManagementApi. A partner must explicitly include this scope to communicate their intent. Without it — even with a valid token — the call will be rejected.

🎤 Analogy — Concert Wristbands (Building on Module 4)

Authentication: You show your ticket at the gate. "Yes, this ticket is real." You're in.

Scopes: Security puts a wristband on you. Different wristbands = different access: General Admission, VIP, or Backstage. The wristband says "this person may request access to these areas." It doesn't automatically open every door.

Authorization: At each door, staff checks: Is this ticket for today? Does this wristband cover this section? Each check is independent.

✏️ Check Your Understanding

A partner has a valid access token but receives a 403 when calling the agency-management endpoint. What is the most likely cause?

ATheir token has expired and they need to re-authenticate
BTheir token was issued without the AgencyManagementApi scope — they didn't request that permission when authenticating
CThe endpoint URL is wrong
DThe JSON payload is missing required fields
✅ Correct! A valid token isn't enough for sensitive endpoints. The token must have been issued with the correct scope. The partner needs to re-authenticate and explicitly request the AgencyManagementApi scope.

Module 7

Reads vs. Writes — One Observes, One Changes Reality

Reading data is observational. Writing data is authoritative. A GET request describes what exists right now. A POST, PUT, or PATCH attempts to change what exists — and payroll reality has consequences.

🔑 Why This Matters at EN

When EN sends a POST to a payroll partner for a deduction update, that partner's system will run that deduction on an employee's next paycheck. Bad writes create lasting, real-world damage. EN enforces correctness up-front — we'd rather stop a bad POST than clean up bad payroll data after the fact.

What Happens When a Write Request Arrives

When EN sends a POST deduction update to a payroll partner, the partner's system checks four layers of validity:

1
Structural Validity

Is the JSON well-formed? Are all required fields present? Are values the correct data types? → If not: 400 Bad Request

2
Referential Validity

Does the referenced resource actually exist? Does this employee exist in payroll? Does this deduction code exist? → If not: 400 or 404

3
Business Validity

Is this deduction allowed for this employee? Does the employee's plan support this code? → If not: 400 Bad Request

4
System Stability

Is the payroll system available and able to process this right now? → If not: 500–504 Server Error

⚠️ The Asymmetry of Reads and Writes

A failed GET is a minor inconvenience — the data simply wasn't returned. A failed write that partially succeeds can corrupt data. A write that succeeds with wrong values causes real financial harm (incorrect paycheck deductions). This is why write operations carry much higher risk levels and are validated far more strictly.

✏️ Check Your Understanding

EN successfully POSTs a deduction update to the payroll partner (receives a 200 response), but the deduction appears on the employee's next check with the wrong amount. What is the most likely cause?

AThe API was down and cached a previous response
CThe POST was accepted but EN's payload contained the wrong perPayAmount value — a 200 means the structure was valid, not that the values were correct
BThe payroll system recalculated the deduction on its own
DThe deduction code was mapped to the wrong plan in EN
✅ Correct! Remember: EN sends deductions TO the payroll partner — it's always EN doing the POST, not the other way around. A 200 response confirms the structure was valid and accepted, not that the values inside were correct. Always verify the payload values match expected amounts before investigating other causes.

Webhooks — Don't Keep Asking, We'll Tell You

APIs are how you ask questions. Webhooks are how systems notify you that something changed. It's the same mechanism — an HTTP call — but with the roles reversed.

Without Webhooks (Polling)

  • Partners must repeatedly call EN asking "did anything change?"
  • High volume of unnecessary API calls
  • Delays — you only know about changes when you ask
  • Wastes compute on both sides

With Webhooks (Push)

  • EN proactively notifies partners when something changes
  • Real-time — the partner is told immediately
  • Efficient — only fires when there's something to report
  • Partner then uses their API to fetch full details
💡 How Webhooks Fit Into EN's Payroll Integrations

When an employee's address changes in a payroll system, payroll sends EN a webhook: "something changed for this employee at this company." EN validates the webhook, authenticates with payroll, then uses a GET call to fetch the full updated record. The webhook is just the notification — the actual data transfer still uses standard API calls.

✏️ Check Your Understanding

A payroll partner says "We sent EN a webhook notification 2 hours ago when an employee was terminated, but EN still shows them as active." What should you check first?

AWhether the employee's deduction code is mapped correctly
BWhether the partner's API subscription is still active
CWhether EN received and processed the webhook — check the Integration Health Dashboard for an incoming demographics change for that employee around that time
DWhether the employee's open enrollment was completed
✅ Correct! The webhook is just a notification. EN may have received it but failed to process the subsequent GET call, or the webhook may not have arrived at all. The Integration Health Dashboard is your first diagnostic tool.

Errors & Debugging — The System Is Telling You Exactly What Went Wrong

API errors aren't mysteries. They're the system telling you precisely which rule was violated. Your job is to interpret them correctly, not just read the message — but classify the failure layer, collect the right evidence, and either fix it or route it correctly.

The Error Layer Reference

Status CodeLayerPlain EnglishTypical Action
401Authentication"I don't know who you are" — token missing, expired, or invalidPartner re-authenticates and gets a new token
403Authorization"I know who you are, but you're not allowed to do this" — wrong scope or permissionCheck scopes; verify the identity has access to this resource
400Request Validity"I can't understand or safely process what you sent" — bad JSON, missing fields, wrong data typesFix the payload structure; check required fields
404Not Found"This resource doesn't exist" — wrong ID or the record was deletedVerify the resource ID exists in the target system
409Conflict"This conflicts with existing data or state" — duplicate record or invalid transitionCheck for duplicates; review the current state of the resource
500Server Error"Something went wrong while I was processing this" — unexpected internal errorRetry later; escalate if persistent
503Unavailable"I'm not able to handle requests right now" — system down or throttledWait and retry with backoff; check partner status page
504Timeout"I didn't finish processing this in time"Retry; investigate if the operation is consistently slow

What to Ask Partners for Every Failure

Standardize what you ask for. This eliminates back-and-forth and gets to root cause faster.

✅ Required Evidence Checklist

1. Endpoint + method — the full URL and whether it was GET, POST, PATCH, etc.
2. Timestamp with timezone — exact time the call was made
3. Company ID + Employee ID — the identifiers used
4. Request headers — especially api-version, auth token, and any custom headers
5. Full request body — the complete JSON payload that was sent
6. Full response — status code AND the full response body (ProblemDetails if present)

✏️ Check Your Understanding

EN POSTs a deduction update to the payroll partner and receives a 400 Bad Request back. The broker says "the payload looks fine to us." What is the correct next step?

ATell them to re-authenticate and try again
BCheck if the payroll system is experiencing downtime
CPull the full request body EN sent and the full response body the partner returned — 400 means the partner rejected EN's payload and the response body will say exactly why
DEscalate to Tier 2 immediately without further investigation
✅ Correct! EN sends deductions to the partner — so when a 400 comes back, it means the partner's system rejected what EN sent. The response body will contain the specific reason (missing field, wrong data type, invalid code, etc.). Always get the full response before escalating.

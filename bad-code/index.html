Step 2: Understand Why — Multi-EIN Connections & Company IDs
This customer has a multi-EIN (subscriber group) setup — meaning their EN integration is linked to more than one company ID in payroll. Deduction codes must exist and be active in every company ID linked to EN, or employees in that company will fail.
🏢
How multi-EIN works: A single EN group can be connected to multiple company IDs in Paycor (or another partner). Think of it as one EN account talking to two or more separate payroll companies. A code set up in Company A doesn't automatically exist in Company B.

The two specific reasons this happens (both are I&I ACC error type)
❌ Cause 1: Code doesn't exist in all company IDs
The deduction code ACC was created in Company A's payroll setup but was never added to Company B. EN tries to send it to both — Company B rejects it because the code isn't there at all.
⚠️ Cause 2: Code exists but is inactive in some company IDs
The code ACC exists in Company B's payroll system but has been marked inactive. EN sends the deduction, payroll rejects it because the code isn't currently active for that company.

📋 Step-by-step resolution for the customer
  • Log into payroll (Paycor or the relevant partner) and navigate to the deduction code setup for each company ID linked to EN.
  • Find code ACC in the company ID where employees are failing (Company B in this example).
  • If the code is missing: create it with the same code value as it exists in the working company ID.
  • If the code is inactive: re-activate it in that company ID.
  • Repeat this for every company ID linked to EN — not just the one that's currently failing.
  • Return to Integration Health → "Needs Action" → select the failed employees → tri-dot icon → Push to Payroll (or use the bulk Push & Pull button).
  • Employees should move to "In Sync" once the push succeeds.
⚠️
Important: The customer must fix the code in payroll before retrying. Simply clicking "Push to Payroll" without fixing the underlying issue will result in the same failure again.
Step 4: Knowledge Check
Answer all three questions to complete the training.

1 A customer says code ACC is failing for some employees but not others. What's the most likely root cause?

✅ Correct! Selective failure on a single code almost always indicates a multi-company ID issue. The code is valid — it just doesn't exist or isn't active in every linked company.
❌ Restarting the integration would affect all employees, not just a subset. The selective pattern points to a company ID configuration gap.
❌ If it were a different benefit enrollment, you'd expect different codes. The same code ACC failing for some and not others is a company ID issue, not an enrollment difference.

2 The customer has fixed code ACC in Company B's payroll. What should they do next to get these employees to "In Sync"?

✅ Correct! After fixing the code in payroll, the customer can immediately retry using the Actions icon → Push to Payroll, or select all affected employees and use the bulk Push & Pull button. No waiting or Support call needed.
❌ "Needs Action" items won't auto-resolve — they require a manual action from the customer. After fixing the code in payroll, the customer must use Push to Payroll to retry.
❌ The customer can self-resolve entirely from the Integration Health dashboard. The Instructions column and Actions menu are specifically designed to eliminate the need for a Support call.

3 The customer asks: "I fixed the code in Company B — is that enough?" What do you tell them?

✅ Correct! The instructions specifically say "every company ID that you have in payroll." If they have three linked company IDs and only fix two, the third will still fail. They need to audit all linked company IDs.
❌ EN doesn't automatically propagate code setup between company IDs — that's managed entirely within the payroll system. The customer must ensure the code is correct in each company ID individually.
❌ The code mapping in EN itself isn't the issue — the code is correctly mapped. The problem is the code's existence/status in the payroll company IDs, which is fixed in payroll, not in EN.
